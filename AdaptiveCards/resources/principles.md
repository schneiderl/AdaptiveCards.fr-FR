# <a name="motivations-and-principles"></a><span data-ttu-id="ce46c-102">Motivations et principes</span><span class="sxs-lookup"><span data-stu-id="ce46c-102">Motivations and Principles</span></span>

## <a name="motivations-behind-the-format"></a><span data-ttu-id="ce46c-104">Motivations du format</span><span class="sxs-lookup"><span data-stu-id="ce46c-104">Motivations behind the format</span></span>

<span data-ttu-id="ce46c-105">Début 2016, plusieurs équipes de Microsoft (notamment Outlook, Windows et Bot Framework) se sont rendu compte qu’elles souhaitaient toutes quelque chose d’extrêmement similaires (« cartes ») et que chacune d’elles concevait indépendamment ses propres solutions :</span><span class="sxs-lookup"><span data-stu-id="ce46c-105">In early 2016, several teams at Microsoft (including Outlook, Windows and the Bot Framework) came to the realization that they all wanted something extremely similar ("cards") and that each of them were designing their own solutions independently:</span></span>

- <span data-ttu-id="ce46c-106">Windows avait son propre format de vignettes et de notifications dynamiques.</span><span class="sxs-lookup"><span data-stu-id="ce46c-106">Windows had its own Live Tiles and Notifications format</span></span>
-  <span data-ttu-id="ce46c-107">Bot Framework utilisait un ensemble de modèles de cartes prédéfinis parmi lesquels les développeurs pouvaient choisir lors de l’envoi de messages de bot.</span><span class="sxs-lookup"><span data-stu-id="ce46c-107">The Bot framework was using a set of predefined card templates developers could choose from when sending Bot messages</span></span>
- <span data-ttu-id="ce46c-108">Outlook utilisait son propre format MessageCard pour sa fonctionnalité de messages actionnables.</span><span class="sxs-lookup"><span data-stu-id="ce46c-108">Outlook was using its own MessageCard format for its Actionable Messages feature</span></span>

<span data-ttu-id="ce46c-109">En même temps, d’autres plateformes telles que LINE, FaceBook Messenger, Slack, et ainsi de suite, définissaient leur propre format de « carte » propriétaire.</span><span class="sxs-lookup"><span data-stu-id="ce46c-109">At the same time, other platforms such as LINE, FaceBook Messenger, Slack and more were defining their own, proprietary "card" format.</span></span> <span data-ttu-id="ce46c-110">Quelques employés de Microsoft se sont donc réunis et ont commencé à tenter de définir un format de carte unique et ouvert, ainsi qu’un ensemble de kits SDK qui :</span><span class="sxs-lookup"><span data-stu-id="ce46c-110">So a few Microsoft employees gathered up and started an effort to define a single, open card format and a set of SDKs that:</span></span>

- <span data-ttu-id="ce46c-111">Faciliteraient l’échange de cartes entre les hôtes.</span><span class="sxs-lookup"><span data-stu-id="ce46c-111">Would facilitate the interchange of cards between hosts</span></span>
- <span data-ttu-id="ce46c-112">Permettraient à chaque hôte de conserver le contrôle du style des cartes afin de garantir la cohérence visuelle.</span><span class="sxs-lookup"><span data-stu-id="ce46c-112">Would allow each host to keep control over the styling of cards to ensure visual consistency</span></span>
- <span data-ttu-id="ce46c-113">Permettraient à une application hôte d’afficher facilement des cartes avec un minimum d’effort par le biais de kits SDK prêts à l’emploi.</span><span class="sxs-lookup"><span data-stu-id="ce46c-113">Would make it easy for a host application to display cards with minimum effort via ready-to-use SDKs</span></span>
- <span data-ttu-id="ce46c-114">Fourniraient également de la valeur à des tiers et finiraient par être largement adoptés par le secteur.</span><span class="sxs-lookup"><span data-stu-id="ce46c-114">And that would also provide value to third parties and eventually get adopted widely by the industry</span></span>

## <a name="principles-governing-adaptive-cards"></a><span data-ttu-id="ce46c-115">Principes régissant les cartes adaptatives</span><span class="sxs-lookup"><span data-stu-id="ce46c-115">Principles governing Adaptive Cards</span></span>

1.  <span data-ttu-id="ce46c-116">**La carte adaptative est un format de carte _simple_ et _déclaratif_**</span><span class="sxs-lookup"><span data-stu-id="ce46c-116">**Adaptive Card is a _simple_ and _declarative_ card format**</span></span>

    1.  <span data-ttu-id="ce46c-117">Elle n’est pas censée être un remplacement ou une alternative à HTML ou XAML.</span><span class="sxs-lookup"><span data-stu-id="ce46c-117">It is not meant as an HTML or XAML replacement/alternative</span></span>
    2.  <span data-ttu-id="ce46c-118">Il n’y a **pas de « code-behind »** avec les cartes adaptatives.</span><span class="sxs-lookup"><span data-stu-id="ce46c-118">There is **no "code behind"** with Adaptive Cards</span></span>
        1. <span data-ttu-id="ce46c-119">Les auteurs de cartes ne peuvent pas incorporer de code personnalisé/arbitraire avec leurs charges utiles. Ainsi, un hôte de carte adaptative n’a jamais besoin d’exécuter de code tiers.</span><span class="sxs-lookup"><span data-stu-id="ce46c-119">Card authors cannot embed custom/arbitrary code with their payloads, and as a result an Adaptive Card host never needs to run third party code</span></span>
        2. <span data-ttu-id="ce46c-120">Le dynamisme et l’interactivité sont obtenus uniquement par le biais de balisage déclaratif.</span><span class="sxs-lookup"><span data-stu-id="ce46c-120">Dynamism and interactivity are achieved solely via declarative markup</span></span>
    3.  <span data-ttu-id="ce46c-121">Cela garantit que l’effort nécessaire pour créer un SDK de carte adaptative pour une nouvelle plateforme demeure raisonnable.</span><span class="sxs-lookup"><span data-stu-id="ce46c-121">This ensures that the effort necessary to build an Adaptive Card SDK for a new platform remains reasonable</span></span>

2.  <span data-ttu-id="ce46c-122">**Le format carte adaptative ne peut pas imposer l’utilisation d’une technologie sous-jacente particulière.**</span><span class="sxs-lookup"><span data-stu-id="ce46c-122">**The Adaptive Card format can't impose the use of any particular underlying technology**</span></span>

    1.  <span data-ttu-id="ce46c-123">Le format de carte adaptative ne repose pas sur JavaScript, C#, Python ou tout autre langage.</span><span class="sxs-lookup"><span data-stu-id="ce46c-123">The Adaptive Card format does not rely on JavaScript, C#, Python or any other language</span></span>
    2.  <span data-ttu-id="ce46c-124">De même, il ne repose pas sur HTML ou XAML, ni sur tout autre framework graphique/d’interface utilisateur.</span><span class="sxs-lookup"><span data-stu-id="ce46c-124">Similarly it doesn't rely on HTML or XAML or any other graphic/UI framework</span></span>
    3.  <span data-ttu-id="ce46c-125">Ainsi, les cartes adaptatives peuvent être rendues de manière native sur n’importe quelle plateforme tant qu’il existe un renderer.</span><span class="sxs-lookup"><span data-stu-id="ce46c-125">This way, Adaptive Cards can be rendered natively on any platform for as long as a renderer exists</span></span>

3.  <span data-ttu-id="ce46c-126">**Le format de carte adaptative est une _propriété partagée_** .</span><span class="sxs-lookup"><span data-stu-id="ce46c-126">**The Adaptive Card format is a _shared property_**</span></span>

    1.  <span data-ttu-id="ce46c-127">Le format et ses kits SDK doivent être open source, et c’est la communauté qui doit être à l’origine de son évolution.</span><span class="sxs-lookup"><span data-stu-id="ce46c-127">Along with its SDKs, the format is to be open source and its evolution community driven</span></span>
    2.  <span data-ttu-id="ce46c-128">Le format n’est donc pas détenu ni piloté par une équipe spécifique.</span><span class="sxs-lookup"><span data-stu-id="ce46c-128">The format is therefore not owned nor driven by any one team</span></span>

4.  <span data-ttu-id="ce46c-129">**Le format de carte adaptative n’est pas conçu « uniquement pour une utilisation par Microsoft »** .</span><span class="sxs-lookup"><span data-stu-id="ce46c-129">**The Adaptive Card format is not designed "just for Microsoft's use"**</span></span>

    1.  <span data-ttu-id="ce46c-130">Au lieu de cela, il est conçu pour répondre aux besoins d’un large éventail d’applications et de cas d’usage.</span><span class="sxs-lookup"><span data-stu-id="ce46c-130">Instead it is designed to suit the needs of a wide variety of applications and use cases</span></span>

5.  <span data-ttu-id="ce46c-131">**Le _groupe de travail de carte adaptative_ régit l’évolution du format**.</span><span class="sxs-lookup"><span data-stu-id="ce46c-131">**The _Adaptive Card Working Group_ governs the evolution of the format**</span></span>

    1.  <span data-ttu-id="ce46c-132">Ce groupe de travail se compose d’un ensemble d’employés de Microsoft qui sont tous impliqués dans la réussite du format.</span><span class="sxs-lookup"><span data-stu-id="ce46c-132">This working group is comprised of a set of Microsoft employees that are all deeply involved in the success of the format</span></span>
    2.  <span data-ttu-id="ce46c-133">Les membres du groupe de travail participent à des réunions hebdomadaires (actuellement le lundi) au cours desquelles ils examinent les propositions de fonctionnalités, discutent des problèmes ouverts et suivent l’avancement global des éléments de travail vNext.</span><span class="sxs-lookup"><span data-stu-id="ce46c-133">The Working Group conducts weekly meetings (currently on Monday) during which feature proposals are reviewed, open issues are discussed and triaged and overall progress on vNext work items is tracked</span></span>
    3.  <span data-ttu-id="ce46c-134">Le groupe de travail utilise les commentaires fournis par la communauté globale, notamment les équipes Microsoft internes, pour décider de l’évolution du format.</span><span class="sxs-lookup"><span data-stu-id="ce46c-134">The Working Group uses feedback from the community at large, including internal Microsoft teams, to decide how the format evolves</span></span>
        1. <span data-ttu-id="ce46c-135">Tout le monde peut participer au groupe de travail en ouvrant des problèmes dans GitHub (voir ci-dessous).</span><span class="sxs-lookup"><span data-stu-id="ce46c-135">Anyone can participate in the working group through opening issues in GitHub (see below)</span></span>
        2. <span data-ttu-id="ce46c-136">Ce sont les problèmes/demandes de fonctionnalités enracinés dans l’utilisation réelle du framework des cartes adaptatives (en tant qu’hôte ou en tant qu’auteur de carte) qui ont le plus d’impact sur l’avenir du format.</span><span class="sxs-lookup"><span data-stu-id="ce46c-136">Issues/feature requests rooted in actual usage of the Adaptive Cards framework (either as a host or as a card author) have the most impact on the future of the format</span></span>
    4.  <span data-ttu-id="ce46c-137">Pour être approuvées par le groupe de travail, les nouvelles fonctionnalités proposées :</span><span class="sxs-lookup"><span data-stu-id="ce46c-137">To be approved by the Working Group, proposed new features:</span></span>
        1. <span data-ttu-id="ce46c-138">Doivent être justifiées par des cas d’usage de la vie réelle.</span><span class="sxs-lookup"><span data-stu-id="ce46c-138">Must be justified by real life use cases</span></span>
        2. <span data-ttu-id="ce46c-139">Doivent avoir une spécification fonctionnelle.</span><span class="sxs-lookup"><span data-stu-id="ce46c-139">Must have a functional specification</span></span>
    5.  <span data-ttu-id="ce46c-140">Les nouvelles fonctionnalités approuvées sont ajoutées au backlog et prises en compte pour vNext.</span><span class="sxs-lookup"><span data-stu-id="ce46c-140">Approved new feature are added to the backlog and considered for vNext</span></span>
        1. <span data-ttu-id="ce46c-141">Les critères utilisés pour hiérarchiser les nouvelles fonctionnalités incluent l’ampleur des scénarios couverts par la fonctionnalité, sa complexité/facilité de maintenance, et bien plus encore.</span><span class="sxs-lookup"><span data-stu-id="ce46c-141">The criteria used to prioritize new features include the breadth of scenarios the feature enables, its complexity/maintainability and more</span></span>
        2. <span data-ttu-id="ce46c-142">En cas de doute, elles sont ignorées.</span><span class="sxs-lookup"><span data-stu-id="ce46c-142">When in doubt, keep it out!</span></span> <span data-ttu-id="ce46c-143">Il est beaucoup plus facile d’introduire ultérieurement une fonctionnalité bien conçue que de vivre indéfiniment avec une erreur.</span><span class="sxs-lookup"><span data-stu-id="ce46c-143">It's a lot easier to introduce a well designed feature later than to live with a mistake forever.</span></span>
    6.  <span data-ttu-id="ce46c-144">Toutes les nouvelles fonctionnalités sont implémentées dans tous les kits SDK pris en charge.</span><span class="sxs-lookup"><span data-stu-id="ce46c-144">All new features are implemented in all supported SDKs</span></span>
    7.  <span data-ttu-id="ce46c-145">Toutes les nouvelles fonctionnalités sont documentées et associées à une carte de test publiée dans le dossier d’exemples.</span><span class="sxs-lookup"><span data-stu-id="ce46c-145">All new features are documented and associated with a test card published in the samples folder</span></span>
    8.  <span data-ttu-id="ce46c-146">Les nouvelles versions du format et des kits SDK passent par une phase bêta.</span><span class="sxs-lookup"><span data-stu-id="ce46c-146">New versions of the format and SDKs go through a beta phase</span></span>
    9.  <span data-ttu-id="ce46c-147">Le planning de publication du schéma de carte adaptative et des versions de SDK est motivé par la qualité, pas par la date.</span><span class="sxs-lookup"><span data-stu-id="ce46c-147">The release schedule for Adaptive Card schema and SDK versions is driven by quality, not date</span></span>

6.  <span data-ttu-id="ce46c-148">**Interopérabilité**</span><span class="sxs-lookup"><span data-stu-id="ce46c-148">**Interoperability**</span></span>
    1.  <span data-ttu-id="ce46c-149">Les cartes créées conformément au format documenté (par exemple celles n’utilisant aucune extension propre à l’hôte) s’afficheront correctement dans n’importe quel hôte qui prend en charge les cartes adaptatives.</span><span class="sxs-lookup"><span data-stu-id="ce46c-149">Cards authored in accordance with the documented format (e.g. not using any host-specific extensions) will render properly in any Adaptive Card-capable host</span></span>
    2.  <span data-ttu-id="ce46c-150">Les seules exceptions à ces principes sont les suivantes :</span><span class="sxs-lookup"><span data-stu-id="ce46c-150">The only exceptions to this principles are:</span></span>
        1.  <span data-ttu-id="ce46c-151">Certains hôtes sont susceptibles de ne pas autoriser l’interactivité et n’afficheront donc pas les entrées et les actions.</span><span class="sxs-lookup"><span data-stu-id="ce46c-151">Some hosts might not allow interactivity, and will therefore not render inputs nor actions</span></span>
        2.  <span data-ttu-id="ce46c-152">L’exécution des actions Action.Submit est à la discrétion de l’hôte, et tous les hôtes ne gèreront pas nécessairement toutes les charges utiles Action.Submit.</span><span class="sxs-lookup"><span data-stu-id="ce46c-152">Execution of Action.Submit actions is at the discretion of the host, and not all hosts will necessarily properly handle all Action.Submit payloads.</span></span> <span data-ttu-id="ce46c-153">De plus, certains hôtes sont susceptibles de ne pas prendre en charge Action.Submit.</span><span class="sxs-lookup"><span data-stu-id="ce46c-153">Furthermore, some hosts might not support Action.Submit at all</span></span>

7.  <span data-ttu-id="ce46c-154">**Le format doit être extensible.**</span><span class="sxs-lookup"><span data-stu-id="ce46c-154">**The format needs to be extensible**</span></span>

    1.  <span data-ttu-id="ce46c-155">Les hôtes doivent avoir la liberté d’ajouter la prise en charge d’éléments personnalisés ou d’actions personnalisées qui vont au-delà des capacités du format.</span><span class="sxs-lookup"><span data-stu-id="ce46c-155">Hosts should have the freedom of adding support for custom elements or custom actions that go beyond what the format is capable of</span></span>

    3.  <span data-ttu-id="ce46c-157">Ces ajouts sont à la discrétion de l’hôte.</span><span class="sxs-lookup"><span data-stu-id="ce46c-157">These additions are at the discretion of the host</span></span>

        3. <span data-ttu-id="ce46c-160">Ils peuvent toutefois être présentés au groupe de travail et proposés en tant que nouvelles fonctionnalités pour une version ultérieure du format, comme décrit au point n°5.</span><span class="sxs-lookup"><span data-stu-id="ce46c-160">They can however be presented to the Working Group and proposed as new features for a future version of the format, as described in point #5</span></span>

Ainsi, les cartes semblent être des extensions natives de l’expérience de l’application.</span><span class="sxs-lookup"><span data-stu-id="ce46c-162">Host apps impose their styling so cards look and feel like they are native extensions of the app's experience</span></span>
    2.  <span data-ttu-id="ce46c-163">Les auteurs de cartes peuvent toujours spécifier le style, mais uniquement par le biais d’expressions sémantiques de couleurs, de tailles, et ainsi de suite.</span><span class="sxs-lookup"><span data-stu-id="ce46c-163">Card authors can still specify styling, but only via semantic expressions of colors, sizes, etc.</span></span>

9.  <span data-ttu-id="ce46c-164">**Des kits SDK seront fournis pour les plateformes de développement les plus connues.**</span><span class="sxs-lookup"><span data-stu-id="ce46c-164">**SDKs will be provided for the most popular developer platforms**</span></span>

    1.  <span data-ttu-id="ce46c-165">Les kits SDK facilitent le rendu des charges utiles de cartes adaptatives dans n’importe quel hôte.</span><span class="sxs-lookup"><span data-stu-id="ce46c-165">SDKs make it easy to render Adaptive Card payloads in any host</span></span>
    2.  <span data-ttu-id="ce46c-166">Cela garantit que la barrière à l’entrée est la plus basse possible à la fois pour les développeurs tiers et pour les équipes Microsoft.</span><span class="sxs-lookup"><span data-stu-id="ce46c-166">This ensures the barrier to entry is as low as can be both for third party developers and Microsoft teams</span></span>
